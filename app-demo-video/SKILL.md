# App Demo Video — AI Skill

## 1. When to use / what this produces

Use this skill when the user needs a **scripted demo video** of a Flutter app: a mock
"screen recording" showing a user journey (open app → interact → outcome) inside a
phone bezel, rendered as an MP4. Typical uses: social/demo clips, landing-page loops,
store preview videos.

The output is a deterministic, fully reproducible video generated by one `flutter test`
run plus one ffmpeg command. Real app widgets over seeded in-memory data — no emulator,
no screen recorder, no manual interaction, no production code changed. A ~24 s 30 fps
video of ~715 frames captures in about a minute.

This skill is the **animation sibling of `app-store-screenshots`**: same bootstrap,
same seeding philosophy, same bezel aesthetic. If that skill is available, reuse its
§8 (font + bootstrap gotchas) and §9 (seeding) wholesale; this skill only repeats the
deltas.


## 2. The core method

A `testWidgets` test pumps the widget tree **one video frame at a time** and captures a
PNG after every pump:

```
tick():
  await tester.pump(33.3ms)          // advances ALL animations deterministically
  boundary.toImage(pixelRatio: 2.0)  // RepaintBoundary at the tree root
  write frame_NNNN.png
```

Everything runs inside one `tester.runAsync(...)` block: real async (file I/O,
`toImage`, DB reads) resolves between ticks, while `tester.pump(duration)` still drives
the fake animation clock. This combination is what makes the whole approach work —
implicit animations, route transitions, ink ripples, and bottom-sheet slides all
advance exactly one frame per tick, perfectly smooth at any fps, with zero flakiness.

The video is then stitched offline:

```
ffmpeg -framerate 30 -i frames/frame_%04d.png \
  -c:v libx264 -pix_fmt yuv420p -crf 18 -movflags +faststart out.mp4
```

**Parameters that worked**: 30 fps (tick = `Duration(microseconds: 1000000 ~/ 30)`);
**vertical Full HD output (1080×1920, 9:16)** — the universal social/device format —
via a **360×640 logical canvas at `pixelRatio: 3.0`**. Both output dimensions must be
**even** or yuv420p encoding fails. MP4/H.264 has **no alpha channel** — the bezel must
sit on a solid (or gradient) backdrop, never "transparent".

**Design once, shrink-fit to 9:16.** Don't lay out natively at 360×640. Design the
stage at the marketing screenshots' 430×932 and render it into the 9:16 canvas with
`OverflowBox(maxWidth/maxHeight: infinity, alignment: topCenter)` →
`Transform.scale(0.75, alignment: topCenter)` → `SizedBox(430×932)` — the same
"Play export" trick as the screenshot skill (§3 there). Lift the phone ~32 canvas-dp
(subtract `32/0.75` from the design bezel-top) so the whole bezel fits the shorter
canvas; the side strips blend invisibly because the gradient is horizontally uniform.
Coordinate rules that fall out of this:
- `tester.getCenter`/`getRect` report **canvas** (global) coords — position the finger
  overlay at canvas level (outside the transform) and no conversion is ever needed.
- Visibility checks (e.g. "is the Save button above the bezel bottom?") must compare
  against design geometry **multiplied by the scale**.
- Drag deltas are scale-invariant end to end (pointer delta D canvas-px → content
  scrolls D/scale design-dp → moves D canvas-px on screen) — compute scroll distances
  from canvas-rect differences directly.
- Hit-testing through `Transform.scale` works fine as long as the tap targets are
  inside the canvas bounds.


## 3. ★ The Director pattern — one clock, no hidden pumps

A small `Director` class owns the frame counter; **every** script primitive funnels
through `tick()`:

- `wait(duration)` → N ticks.
- `tapOn(finder)` → finger indicator appears (≈250 ms), `startGesture(getCenter(f))`,
  ~130 ms held (ripple grows on screen), `g.up()`, finger fades (≈330 ms).
- `drag(finder, offset, duration)` → `startGesture` + per-tick `moveBy(total/n)`.
- `note(label)` → prints `frame NNNN  <beat>` so the log maps frames to beats.

**The cardinal rule: never call `tester.tap`, `tester.timedDrag`, or `pumpAndSettle`
in the script.** They pump internally without capturing — animation time passes but no
frames are written, so the video visibly "skips". `pumpAndSettle` additionally hangs
forever on indefinite animations. Use only `startGesture` / `moveBy` / `up` plus the
director's own ticks.

The script itself is a plain sequential async function — an imperative beat sheet, not
a declarative action list. State changes (scene switches, overlay toggles) are
`ValueNotifier` flips; the implicit animations they trigger advance frame-by-frame
through subsequent `wait()` calls.


## 4. The stage — scene stack inside a bezel

One widget tree is pumped **once** for the entire video:

```
RepaintBoundary(repaintKey)
└─ outer MaterialApp (app theme + l10n delegates)
   └─ Material(transparency) → Container(color: backdrop)
      └─ Stack
         ├─ Positioned phone bezel (dark rounded Container, ClipRRect)
         │  └─ Column [ blank status bar (28dp),
         │     Expanded(MediaQuery(size: inner, padding: zero) → scene Stack:
         │        1. mock home screen            (bottom layer, always built)
         │        2. _SceneLayer(mock source app)        — e.g. bank receipt
         │        3. _SceneLayer(nested real-app MaterialApp)
         │        4. effect overlays: snap thumbnail, notification card, flash ])
         └─ FingerOverlay (root level — above the bezel)
```

**`_SceneLayer`** = `ValueListenableBuilder<Scene>` → `IgnorePointer(ignoring:
!active)` → `AnimatedOpacity` + `AnimatedScale`. Inactive scale ~0.85–0.92 gives an
"app zooms open" feel when a scene activates. `IgnorePointer` on every inactive layer
and every decorative overlay is **load-bearing**: gestures must always reach the active
scene.

**The nested MaterialApp** hosts the real app screens with its own `navigatorKey` so
the script can push routes directly. Give it the app's real theme **plus**
`splashFactory: InkRipple.splashFactory` — Material 3's default `InkSparkle` uses
fragment shaders that don't render in `flutter test`, so without the override taps
show no ripple at all.

**MediaQuery override** on the bezel interior (size = inner content, paddings zero)
keeps modal bottom sheets and dialogs sized to the bezel. Note `showModalBottomSheet`
defaults to `useRootNavigator: false`, so sheets opened by real screens stay inside the
nested app — verify any custom sheet helper doesn't pass `useRootNavigator: true`.

**The bezel** is drawn in Dart (no asset): `Color(0xFF1A1A1E)`, radius 38, 1.5 dp
gloss border (white ~13%), soft drop shadow, blank surface-colored status-bar strip
(no clock/icons — they'd render as boxes and add noise).

**Brand-match the canvas to the marketing screenshots.** The strongest version of the
video reuses the screenshot frames' exact visual system: the same emerald gradient on
the canvas, the same headline caption above the bezel (display font, w700, white,
34 dp design, natural word wrap — never forced `\n`), and the same 344×672 bezel with
0.82 text scale inside. The video then reads as the animated sibling of the store
screenshots.

**End on a CTA outro.** After the in-app confirmation beat, fade in a full-canvas
outro over everything (gradient + logo + app name + one line like "Start your free
trial today", ~600 ms fade, ~2.5 s hold). ⚠ A full-canvas overlay that is a direct
child of a `Stack` **must be wrapped in `Positioned.fill`** — a non-positioned child
becomes the Stack's size reference and silently shifts every other layer's geometry
(this produced a completely garbled composition before the fix).


## 5. ★ Real interactions, visible feedback

The believability of the video comes from three layers working together:

1. **Real gestures** — `startGesture` → hold a few ticks → `up()` drives the actual
   `InkWell`/button, so the press ripple, the bottom-sheet slide, the route transition
   are all the app's genuine behavior captured frame-by-frame.
2. **Synthetic finger indicator** — a ~38 dp translucent circle (black ~22% fill,
   white60 2 dp border, soft shadow) positioned at `tester.getCenter(finder)`. It fades
   in before the press, shrinks to ~0.82 scale while pressed, fades out after. Implement
   as a StatefulWidget that remembers the last position so the fade-out doesn't snap to
   `Offset.zero`. Keep it at the **root** stack level (above the bezel) wrapped in
   `IgnorePointer`.
3. **Finder hygiene** — find by visible text exactly as the screen renders it (grep the
   l10n files for the real strings), `Key`s for bespoke mock elements, and
   `.hitTestable()` when the same text could exist behind an open sheet. A tap target
   inside a decorative overlay (e.g. the notification card) needs no real handler — the
   finger taps it visually while the director performs the transition itself.


## 6. ★ The invisible pre-push trick

Problem: pushing a real data-loading screen at the moment of the on-video tap shows its
loading spinner for several frames.

Solution: while a *previous* beat is still on screen (and the nested-app layer is at
opacity 0), push the route through the `navigatorKey` and wait ~700 ms of ticks. The
screen loads, the push animation completes — all invisible. When the user "taps", the
director just flips the scene notifier and the fully-loaded screen fades in.

Bonus: push with a regular `MaterialPageRoute` (not a zero-duration route). The push
animation plays unseen, but the eventual **pop after Save plays its real reverse
transition visibly**, revealing whatever sits beneath in the nested navigator — make
that a purpose-built end card (brand color, check mark, "Transaction saved" summary)
and the save confirmation beat comes for free.

Pass route arguments exactly like the production app does (e.g.
`RouteSettings(arguments: ReviewArgs(txnId: ...))`) — screens that read
`ModalRoute.settings.arguments` work unmodified.


## 7. Effect overlays

All overlays live in the scene stack above the app layers, each driven by one
`ValueNotifier<bool>` + implicit animations, each wrapped in `IgnorePointer`.

- **Screenshot flash**: `Positioned.fill` white `ColoredBox` under `AnimatedOpacity`.
  Asymmetric timing without a custom widget: `duration: ms(on ? 80 : 220)` — the
  duration is re-read on rebuild, so pop-in is fast and decay is slow.
- **Snap thumbnail** (Android screenshot chip): miniature replica of the current screen
  in a corner, `AnimatedScale` 1.5→1.0 with `Curves.easeOutBack` + quick fade-in.
  Lingers ~1.6 s, fades before the next beat.
- **Heads-up notification**: app-style notification card sliding from the top —
  `AnimatedSlide` offset `(0,-1.3) → zero`, ~450 ms, `easeOutCubic`, plus fade. Reuse
  the same card widget as the marketing screenshots if one exists.
- **Mock home screen**: a wallpaper gradient + grid of rounded-square fake app icons
  with 10 dp labels + a dock row. Include the source app icon (with a `Key`) and your
  app's real logo widget. ~80 lines; sells the "this is a phone" illusion at frame 0.


## 8. Beat-sheet design & pacing

Write the script as numbered beats with named `ms(...)` constants — pacing is tuned by
editing durations and re-running (captures are fast; ~45 s for 650 frames).

Pacing rules that held up:

- **Hold after every information beat**: 1.4–2.0 s on any screen the viewer must read
  (receipt, pre-filled form). 1.2 s after every state change (sheet opened, field
  updated). Animations are fast; comprehension is slow.
- A tap cycle (finger in → press → release → fade) takes ~0.9 s. Budget it.
- Story arc that works for a capture/automation app: context (home) → source app →
  the magic (capture effect + notification) → the product (real screen, pre-filled)
  → one corrective edit (shows the user is in control) → save → in-app confirmation
  card (~1.8 s) → full-canvas CTA outro (§4) as the final hold.
- Scroll beats must be **conditional and measured**: compare the target's
  `tester.getRect` against the bezel's visible bottom (in canvas coords) and scroll
  only the needed delta. An unconditional scroll when the content already fits reads
  as an ugly overscroll stretch; a missing scroll leaves the button half-clipped.
- Total: 20–25 s. Longer demos lose social-feed viewers.

**Verify the journey is real**: end the test by asserting the DB state the journey
should have produced (e.g. row status = confirmed, categoryId = the picked category).
The video then doubles as an end-to-end test of the real save path.


## 9. Bootstrap, seeding, capture hygiene

Same foundation as `app-store-screenshots` §8–9: in-memory sqflite ffi, channel mocks,
`SharedPreferences.setMockInitialValues`, load **all** fonts from `FontManifest.json`
in `setUpAll`, seed via raw `db.insert` with deterministic category IDs, wipe tables
first. Deltas for video:

- **Seed the journey's own row** (e.g. the pending transaction the demo reviews) with
  values matching the mock source screen exactly — same amount, merchant, account — and
  the state the real pipeline would produce (e.g. category = the AI fallback, so the
  demo's "fix the category" beat tells a true story.)
- **Brand safety — no real bank identities.** Never depict a real bank's app,
  notification, or account label ("Chase Checking ••4521") in marketing assets: it
  implies affiliation, invites trademark complaints, and is a store-review rejection
  surface (Apple 5.2.1, Play IP/impersonation policy). Use obviously fictional names
  with zero real-world collision risk (e.g. **Cat Bank / Meow Bank / PawPay** with a
  custom-painted cat-head icon — not even "MyBank", which real apps use). Merchant
  names inside transaction lists (Whole Foods, Netflix, Uber) are standard fintech
  practice and fine to keep.
- Offline write paths must degrade gracefully: confirm repository writes fall back to
  a local/dirty write when the backend isn't configured before scripting a Save.
- **Wipe the frames dir** at test start (`delete(recursive) + create`) — stale frames
  from a longer previous run would silently extend the stitched MP4.
- **FlutterError suppression + completed flag**: suppress `FlutterError.onError` for
  the run, set `completed = true` as the last statement in `runAsync`, and
  `expect(completed, isTrue)` after — an aborted loop must not pass with partial
  frames. Also assert a minimum frame count (≈ expected − margin); update it when the
  script's length changes.
- Pump once (~50 ms) after `pumpWidget` before frame 0 so first-layout shifts don't
  land in the video.


## 10. Stitching & the runner script

Don't run ffmpeg from inside the Dart test — keep capture and encode separate so you
can re-stitch (tune crf/fps) without a recapture. Ship a small runner script
(`tool/make_demo_video.ps1` or `.sh`) that:

1. Runs `flutter test test/screenshots/demo_video_harness.dart --timeout=600s`
   (skippable via a `-SkipTest` flag for re-stitch-only runs).
2. Resolves ffmpeg: PATH → winget Links dir → winget Packages glob (a fresh
   `winget install Gyan.FFmpeg` isn't on PATH until the shell restarts).
3. Encodes: `-framerate 30 -i frame_%04d.png -c:v libx264 -pix_fmt yuv420p -crf 18
   -movflags +faststart`.

Frames go under `build/` (gitignored); commit the MP4 like the store PNGs. Verify the
result with ffprobe: width/height/fps/nb_frames/duration as expected.

⚠ PowerShell 5.1 gotcha: with `$ErrorActionPreference = 'Stop'` inside the script, do
NOT invoke the script with stderr redirected (`script.ps1 2>&1 | ...`) — ffmpeg writes
its banner to stderr, PS 5.1 wraps redirected native stderr as ErrorRecords, and the
combination turns the banner into a terminating error that kills the encode mid-file
(leaving a stale MP4 that looks current). Run the script plainly.

**Embedding on a landing page**: `<video autoplay muted loop playsinline
preload="metadata" poster="...jpg" width="1080" height="1920">` — `muted` +
`playsinline` are required for mobile autoplay; explicit width/height prevents layout
shift; generate the poster JPEG from frame 0 with ffmpeg and give the element a
brand-color `background` so nothing flashes white before load. A natural placement is
a "How it works" steps section: steps stacked in one column beside the looping video
(video first, capped width, on mobile).


## 11. Gotchas recap (the ones that bite)

- `tester.tap` / `pumpAndSettle` in the script → skipped frames / infinite hang (§3).
- M3 `InkSparkle` → no visible ripples in tests; override with `InkRipple` (§4).
- MP4 has no alpha; odd pixel dimensions break yuv420p (§2).
- Missing `IgnorePointer` on an overlay silently swallows a scripted tap (§4).
- A full-canvas overlay added to a Stack without `Positioned.fill` resizes the Stack
  and shifts every layer (§4).
- Loading spinner caught on camera → invisible pre-push (§6).
- Unconditional scroll beats → overscroll stretch or clipped buttons; always measure
  first (§8).
- Icons as hollow boxes → load all `FontManifest.json` families (screenshot skill §8).
- Stale frames extending the video → wipe the frames dir (§9).
- Real bank names/UI in any frame → trademark + store-review risk; fictional cat-bank
  identities (§9).
- PS 5.1 + `2>&1` + ErrorActionPreference Stop kills ffmpeg on its stderr banner (§10).


## 12. Verification checklist

- [ ] Test passes including the end-state DB assertions and minimum frame count.
- [ ] Spot-check PNGs at every beat boundary (the `note()` log gives frame numbers):
  flash mid-decay, notification fully in, form pre-filled (no spinner), sheet open
  with finger on the item, updated field, end card.
- [ ] ffprobe: expected width/height (even numbers), 30 fps, expected duration.
- [ ] Watch the MP4 end-to-end: pacing feels unhurried, every tap shows finger +
  ripple, no frame jumps, loop point is clean.
- [ ] No PII in any frame; mock account labels only.
- [ ] Re-run produces the same video (dates aside) — determinism is the point.
