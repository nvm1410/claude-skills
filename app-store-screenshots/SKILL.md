# App Store Screenshots — AI Skill

## 1. When to use / what this produces

Use this skill any time the user needs framed, captioned marketing screenshots for a
Flutter app — for Play Console, App Store Connect, or any store listing that needs
multiple formatted images.

The output is a set of PNGs (typically 7–10 per store) where each image shows the app
running inside a clean device bezel, sitting on a branded gradient background, with a
bold caption banner above and one "hero" UI element enlarged and floating outside the
bezel to draw the eye. Everything is generated by a single `flutter test` command; no
manual screenshotting, no Figma, no Photoshop.

What you get:
- ~85% automated: seeded data, real widgets, device frame, caption, hero treatment, both
  store sizes — all from one command.
- Fully reproducible: re-run any time to regenerate all images from source.
- No production code changed: the harness lives in `test/screenshots/` and ships nothing
  into the app binary.


## 2. The core method

A `testWidgets` test writes PNG files as a side effect using
`RepaintBoundary.toImage(pixelRatio: 3.0)`. It is not a correctness test — suppress
`FlutterError.onError` so layout overflows and null-arg warnings don't fail the run.

Each screenshot is composed by a wrapper widget called something like `_ScreenshotFrame`,
which is a `Stack` containing:

1. A full-canvas gradient background (brand color top → darker brand color bottom).
2. A caption area at the top (~20% of canvas height): headline in the brand display font,
   weight 700, white; optional sub-caption at ~72% opacity, weight 400.
3. A device bezel: a dark rounded rectangle (~80% canvas width, centered) that clips its
   child via `ClipRRect`. Inside: a blank status-bar gap at the top, then the real app
   screen rendered via `MediaQuery` text-scale override (see §6).
4. A hero element rendered **after** the bezel in the Stack so it always paints on top.
   It is positioned via `Positioned` (not clipped) and scaled ~1.15× from its top-center
   with a drop shadow, so it visibly overflows the bezel edge.

Wrap the outermost Stack in `Material(type: MaterialType.transparency)` to provide a
`DefaultTextStyle` ancestor; without it Flutter paints a yellow double-underline under
all caption text.


## 3. ★ The "design Apple, shrink-fit Play" trick

**Design everything once at the App Store logical size: 430×932 dp.**

For the Play export (360×640 dp logical), do **not** build a separate smaller layout.
Instead, take the identical 430×932 layout and render it inside:

- `OverflowBox(maxWidth: infinity, maxHeight: infinity, alignment: Alignment.topCenter)`
  — this removes the 360×640 layout constraint so the inner widget can lay out at its
  native 430×932 size.
- `Transform.scale(scale: 0.75, alignment: Alignment.topCenter)` wrapping a
  `SizedBox(430×932)` — scales the full layout down to fit the Play canvas.

Export configs hold `(dir, logicalSize, textScale)`. The capture loop sets the surface
size to the logical size for each config, pumps the same `_ScreenshotFrame`, and writes
to the matching directory.

`pixelRatio: 3.0` on both → 1290×2796 px (App Store 6.7") and 1080×1920 px (Play).

The gradient runs top-to-bottom, so the color at any given y-position is identical
regardless of canvas width. The narrower Play canvas just exposes more gradient on the
sides of the scaled phone — it looks intentional, not broken.


## 4. Output spec

| Store | Logical canvas | px ratio | Physical output |
|-------|---------------|----------|-----------------|
| App Store 6.7" | 430×932 dp | ×3.0 | 1290×2796 px |
| Google Play phone | 360×640 dp | ×3.0 | 1080×1920 px |

- Two output dirs: `screenshots/appstore/` and `screenshots/play/`.
- N frames × 2 stores = 2N files total. Typical range: 7–10 frames.
- Slug-named: `01_ai_capture.png`, `02_transactions.png`, etc.
- Light theme throughout — dark themes are harder to read in thumbnail previews.
- Do not gitignore the output; commit the PNGs so they can be uploaded directly from
  the repo.


## 5. Frame set & ordering principle

**The first three frames carry the pitch.** Most users decide before swiping to frame 4.
Lead with the single hardest-to-replicate differentiator, not a generic dashboard.

Ordering logic:
1. The magic moment (the thing no other app does).
2. The daily-use proof (shows the app is real and complete).
3. The trust/control moment (answers "but can I verify it?").
4–N. Supporting features that reinforce completeness.

**Kachak worked example** (7 frames, AI auto-capture differentiator):

| # | Slug | Headline | Hero element |
|---|------|----------|--------------|
| 1 | `01_ai_capture` | Snap a screenshot. We log the rest. | Android notification card from Kachak floating above a mock bank receipt screen |
| 2 | `02_transactions` | Search and filter every transaction. | Search bar + active filter chip at the top |
| 3 | `03_notifications` | Tracks spending while you sleep. | Double OS notification stack (bank below, Kachak above) + app-picker dialog below |
| 4 | `04_dashboard` | Your whole month, at a glance. | Net balance card (slot hero — replica of the real net card, §7b) |
| 5 | `05_review_queue` | AI captures. You confirm. | Review tile with confirm/dismiss actions |
| 6 | `06_budgets` | Set budgets. See them fill up in real time. | Expanded over-budget parent card with child allocation rows (slot hero) |
| 7 | `07_insights` | See exactly where it goes. | Top-3 spending rows card aligned with the legend below it (slot hero) |

**Rule: one hero per frame.** One element floats outside the bezel. Everything else stays
inside. If you have two things to emphasize, make one the hero and let the other be
visible-but-inside.

**Rule: the hero REPLACES its counterpart.** When the hero is an enlarged copy of an
element that exists on the screen (a summary card, a budget card, top list rows), the
screen must not also show that element — no duplicates — and the enlarged hero must not
cover any sibling section. Use the slot-hero system (§7b) to satisfy both at once.

**Rule: a floating hero over a real list must show a DISTINCT item.** When a floating
hero (§7a) is "one of the list tiles, enlarged" and the real screen behind it still
renders its list, give the hero content that is NOT one of the visible rows (a
plausible extra item) — otherwise the frame reads as a duplicated row. And build the
hero from the app's real shared widgets (e.g. the themed `Card` + the shared
transaction-tile widget with its real trailing buttons), not a hand-styled
approximation — hand-built replicas drift from the real UI (wrong radius, missing
breadcrumb, amount shown where the real screen shows action buttons) and users notice.

**Rule: horizontal overflow only.** The hero is the only element allowed to overflow the
bezel, and only left/right. Nothing overflows the bezel top or bottom; in-bezel content
clipping at the bezel bottom (like a scrolled screen) is fine.

The exception: a composite hero that *contains* multiple sub-elements (e.g. two stacked
notification cards, or a double stack of notifications + a dialog below) still counts as
one hero — it is one positioned/scaled unit in the outer Stack.


## 6. Anatomy of a composed frame

**Caption area** (top ~20% of canvas):
- Headline: brand display font, weight 700, white, ~34 dp, `height: 1.22`, natural word
  wrap — never force `\n` breaks; let the text fill the banner width.
- Sub-caption (optional): same font, weight 400, ~15 dp, `Colors.white.withValues(alpha: 0.72)`.
- Always set `decoration: TextDecoration.none` on caption text — without it the
  `Material(transparency)` ancestor still lets underlines through in some Flutter versions.
- Left-align consistently across all frames.

**Device bezel**:
- Dark rounded rectangle: `Color(0xFF1A1A1E)`, `BorderRadius.circular(38)`, hairline
  border (`Colors.white` at ~13% opacity, 1.5 dp), drop shadow.
- Width: 80% of the 430 dp App Store canvas = 344 dp, centered (43 dp side margins).
- Height: canvas height minus caption area minus bottom margin. **Never resize bezel
  height** for the hero system — let heroes overflow the bottom if needed.
- Status bar: a blank surface-colored strip at the top of the bezel (~28 dp). No clock,
  no signal icons — they render as glyph boxes before you discover the font loading trick
  (§8), and they add noise anyway.
- Inner content: the real app screen inside a `ClipRRect` (radius slightly less than
  bezel radius). Override text scale via `MediaQuery` to ~0.82 so real UI doesn't
  overflow the narrower mock: wrap the child in
  `MediaQuery(data: mq.copyWith(textScaler: TextScaler.linear(0.82)), child: screen)`.

**Hero placement**: always rendered last in the outer Stack with `clipBehavior: Clip.none`
so it overflows freely. See §7.


## 7. ★ The hero "enlarge + overflow" system

One element per frame is "promoted" outside the bezel: rendered in the outer Stack via
`Positioned`, scaled 1.15×, with a `BoxShadow` drop shadow. Because it paints after the
bezel, it is always on top. Because the outer Stack has `clipBehavior: Clip.none`, it
can overflow the canvas edges. Two variants exist:

- **§7a Floating hero** — a bespoke mock (notification card, dialog) that doesn't
  correspond 1:1 to an element on the screen behind it. Positioned by hand-tuned
  fractions.
- **§7b Slot hero** — an enlarged replica of an element the screen normally shows.
  The screen is a composite mock that reserves an empty slot for it; the hero is
  centered into the measured slot. Use this whenever the hero has an on-screen
  counterpart, otherwise you get duplicated content or covered siblings.

### 7a. Floating heroes — three positioning knobs

### Knob 1 — at-top vs centered (simple default)
- `heroAtTop: true` → hero straddles the top edge of the bezel (good for cards that
  should appear to "fly out" of the phone top).
- `heroAtTop: false` (default) → hero is vertically centered in the lower half of the
  bezel (good for bottom-edge overflow like budget bars, spending cards, review tiles).

### Knob 2 — hero fraction
`heroFraction: double?` pins the hero's top edge to:
```
naturalBezelTop + bezelHeight × heroFraction
```
where `naturalBezelTop` is the *un-shifted* bezel top (caption height + gap), regardless
of any bezel-top offset (knob 3). Use `heroFraction: 0.0` to pin the hero flush with the
natural bezel top — the hero floats *above* the phone when the bezel is shifted down.

**Important**: anchor `heroFraction` to the un-shifted bezel top, not the shifted one.
Otherwise the hero moves when you adjust the bezel position.

### Knob 3 — bezel-top offset
`bezelTopOffset: double` shifts the *rendered* bezel down by that many dp (in App Store
coordinate space) without moving the hero. Use this to create a gap between the hero
(pinned above) and the bezel (pushed below), so the hero truly floats in front of the
phone rather than emerging from it.

Intentional canvas bottom overflow is fine — use it when you want to show "partial"
content like a notification emerging from above or a dialog partially cut off at the
bottom.

### 7b. ★ Slot heroes — replace, don't duplicate or hide

When the hero is an enlarged copy of something the screen shows (net-balance card,
budget card, top legend rows), floating it over the real screen fails twice: the real
counterpart is still visible somewhere (duplicate), and the 1.15× extra height covers
siblings. Real screens can't reserve extra space around a widget, so:

1. **Build a composite mock of the screen** (same chrome, sections, exact l10n strings
   — see §10) that renders everything EXCEPT the counterpart. Where the counterpart
   would be, place an empty `SizedBox(key: heroSlotKey, height: slotHeight)` with the
   screen's normal sibling gaps on both sides.
2. **`slotHeight = hero natural height × 1.15`** — the slot absorbs the enlargement, so
   the visible gaps between the enlarged hero and its siblings equal the screen's
   normal spacing. Get the natural height from the run diagnostics (below), not by
   calculation.
3. **Measure the slot at runtime** instead of hand-tuning fractions: pump with the hero
   hidden, `tester.getRect(find.byKey(heroSlotKey))`, then convert global → frame-local
   via a `GlobalKey` on the frame's content Stack and `RenderBox.globalToLocal` (this
   also inverts the Play 0.75 scale transform automatically — works on both passes).
4. **Reveal without re-pumping**: drive the hero layer with a `ValueNotifier<Rect?>` +
   `ValueListenableBuilder` wrapping the `Positioned` (legal — `ParentDataWidget` binds
   through non-RenderObject wrappers). Setting the notifier rebuilds only the hero.
5. **Overlay geometry**: `Positioned(top: slot.top, height: slot.height, left:
   bezelLeft, width: bezelW)` → `OverflowBox(maxHeight: double.infinity, alignment:
   center)` → `Transform.scale(1.15, alignment: center)` → `MediaQuery(textScaler:
   0.82)` → hero. The 0.82 matches the bezel content's text scale, so the hero is a
   **uniform 1.15× enlargement** of its counterpart (text, padding, icons together).
   Pre-scale width = bezelW (344) → 396 visual → ±26 dp horizontal bezel overflow.
   The `OverflowBox` is load-bearing: without it the slot's tight height clamps the
   hero's layout and silently distorts it.
6. **Placement band**: position the slot so the hero's top edge lands within 20–70% of
   the bezel height. Print a band diagnostic each run.

Diagnostics to print per slot frame (App Store pass): slot height, hero natural size ×
1.15, and the band percentages, e.g.
`04_dashboard slot=151dp hero=151dp (natural 131×344) band=32–55%`.
When `hero` ≠ `slot`, update the mock's `slotHeight` constant to the printed hero value.
Use `tester.getSize` for the natural size (layout size, transform-free); `tester.getRect`
composes ancestor transforms — right for global-rect measurement, wrong for natural size.

**Don't try to measure/tap the real screen instead of mocking.** A first attempt that
kept the real screens and aligned heroes onto measured counterparts worked, but needed
`scrollUntilVisible` (stops at 1-pixel visibility), `warnIfMissed: false` taps (silently
miss), and expand-toggle choreography — and it still couldn't create space around the
counterpart, so the enlarged hero always ate the sibling gaps. The slot mock is less
clever and strictly better.

### Combined hero (multiple elements, one positioned unit)
When a frame needs two elements at different vertical positions (e.g. notification cards
at the top AND an app-picker dialog lower down), wrap them in a `SizedBox` with an
explicit height containing a `Stack(clipBehavior: Clip.none)` with `Positioned` children
inside. The whole unit is still one hero — one `Positioned` with one `Transform.scale`
in the outer Stack. Size the inner SizedBox tall enough to contain both children; canvas
overflow at the bottom is acceptable.

To convert "where in the bezel" to "pre-scale top offset within the hero SizedBox":
```
hero_internal_top = (target_canvas_y - heroY) / 1.15
```
where `heroY` is the canvas y of the hero's `Positioned.top`, and `1.15` is the scale factor.

### Dialog / popup heroes
When the hero is a dialog or bottom sheet:
- Constrain its width to ~80% of bezel width (344 × 0.80 = 275 dp visual). Pre-scale
  width = 275 / 1.15 ≈ 240 dp. Use `Align(topCenter, child: SizedBox(width: 240, ...))`.
- Add a modal scrim inside the *screen body* (not the outer Stack): a `Positioned.fill`
  `ColoredBox` with ~45% black opacity renders behind the outer-Stack hero automatically,
  making the dialog look exactly like a real `AlertDialog` with a dimmed background.
- `Color(0x72000000)` (hex 72 = ~45% alpha) is a good starting opacity for the scrim.


## 8. Font + bootstrap gotchas

### Every icon renders as a hollow box
**Cause**: only the brand font is loaded. `MaterialIcons` and `CupertinoIcons` glyphs
fall back to the Ahem box font.  
**Fix**: load *all* font families from `FontManifest.json` before the test body runs.
Pattern (adapted from `golden_toolkit`'s `loadAppFonts`): read `FontManifest.json` via
`rootBundle.loadString`, iterate the family list, create a `FontLoader` per family,
call `addFont(rootBundle.load(assetPath))` for each asset in the family, then
`await loader.load()`. Run all loaders in parallel with `Future.wait`. `flutter_svg`
(if your app uses it for the logo) renders vector paths and needs no font.

### Yellow double-underline on caption text
**Cause**: `Text` widgets in the outer `Stack` have no `Material`/`Scaffold` ancestor.  
**Fix**: wrap the outermost `Stack` in `Material(type: MaterialType.transparency)`.
Belt-and-suspenders: also set `decoration: TextDecoration.none` on every caption/hero
`TextStyle`.

### Bootstrap sequence
Run in `setUpAll` (before the test body):
1. `sqfliteFfiInit()` + `databaseFactory = databaseFactoryFfi` — in-memory SQLite.
2. Mock the permissions method channel (return `granted` for all checks).
3. Mock the app's control method channel (return null for all calls).
4. `SharedPreferences.setMockInitialValues(...)` — include `serviceEnabled: true`,
   `setupGuideSeen: true`, `introSeen: true`, and any other flags the app reads on init
   to avoid redirect/setup flows appearing in screenshots.
5. Open all repositories, init currency service, seed default categories.

### Settle timing
After `pumpWidget`, do not call `pumpAndSettle()` if any widget runs an animation that
never settles (e.g. `LinearProgressIndicator` running indefinitely). Instead:
1. `await Future.delayed(Duration(milliseconds: 700))` — lets progress-bar animations
   complete their initial fill.
2. `await tester.pump(Duration.zero)` — flushes microtasks.
3. `await tester.pump(Duration(seconds: 3))` — advances time for any remaining
   animations.
Then capture. After revealing a slot hero (notifier fire), pump ≥350 ms again so the
hero's own progress-bar tweens finish.

### Silent abort under FlutterError suppression
With `FlutterError.onError` suppressed, an exception inside `tester.runAsync` can abort
the capture loop mid-way while the test still reports PASS — leaving stale PNGs on disk
that look current. Guard it: set a `completed = true` flag as the last statement inside
the `runAsync` closure and `expect(completed, isTrue)` after it.


## 9. Seeding mock data

Seed directly via raw `db.insert(table, row)` calls — do not go through the app's
repository write-through path (which requires a live backend). Always wipe tables first:
```
await db.delete('transactions');
await db.delete('budgets');
```
This makes totals deterministic no matter how many times the harness has run in the
same process session.

**Category IDs**: most apps have deterministic category IDs (e.g. derived from a hash of
`userId + kind + name`). Use the same formula for seeded rows so they attach to the
real categories displayed in the UI. Do not use random UUIDs for categories.

**Persona design**:
- Net-positive month: income > expenses so the hero balance card reads as a win.
- At least one "over budget" category — the red over-limit state is the most
  recognizable budgeting moment and makes for a compelling screenshot.
- 3–5 items in the review queue: enough to fill the list without looking sparse.
- Merchants: generic, globally recognizable (Whole Foods, Amazon, Netflix, Uber) — not
  regional brands that half your audience won't recognize.
- No PII: no real account numbers, no real names.

**Brand safety — no real bank identities.** Merchant names inside transaction lists
are standard fintech practice and fine. But never depict a real bank's product —
a fabricated "Chase Bank" push notification, a Chase-labeled receipt screen, or a
`Chase Checking ••4521` account label — in marketing assets: it implies affiliation,
invites trademark complaints from heavily-policed marks, and is a store-review
rejection surface (Apple 5.2.1, Play IP/impersonation policy). Use obviously
fictional names with zero real-world collision risk — e.g. **Cat Bank / Meow Bank /
PawPay**, account label `Cat Bank Checking ••4521`, with a custom-painted cat-head
icon (CustomPaint: face circle + two ear triangles + `BlendMode.clear` eye holes in a
saveLayer) instead of a generic bank glyph. Avoid even plausible names like "MyBank"
— real apps use them. Prominent items in a review-queue/list screenshot read better
as generic actions anyway ("Morning coffee", "Buy new clothes") than as brand rows.

**Kachak persona** (worked example):
- Income: $4,000 salary + $250 freelance = $4,250.
- Expenses: ~$2,830 across 8 categories; Shopping at $521 vs $400 budget = 130% over.
- Review queue: Morning coffee −$5.75, Lunch delivery −$24.80, Buy new clothes
  −$138.40 (3 items, generic actions — no brands); the frame's floating hero is a
  distinct 4th item, "Buy a new phone" −$329.00.


## 10. Composite frames need bespoke fakes

When the frame isn't simply "a real screen in the bezel" — e.g. an AI-capture composite,
a notification-settings screen with an OS notification floating above, or a settings
screen with a dialog hero — you build lightweight mock widgets that *look* like the real
thing.

**Patterns worth reusing:**

- **Slot composites for replacement heroes** (frames 4/6/7 in Kachak): mock the whole
  screen — app bar, month picker, summary header, sibling cards/rows — around an empty
  hero slot (§7b). Reuse the app's real public widgets directly (`StatCard`,
  `MoneyText`, `KachakProgressBar`, `CategoryAvatar`, `MonthPicker`, `SectionHeader`,
  real `SegmentedButton`, `fl_chart` `PieChart` with the same section colors) and grep
  the real screen for exact l10n strings, paddings, and text styles. Share row/card
  replica widgets between the hero and its in-mock siblings so styling can't diverge.
  Use `ListView(physics: NeverScrollableScrollPhysics())` for mock bodies — a `Column`
  that exceeds the bezel paints debug overflow stripes into the PNG; a ListView clips
  cleanly like a real scrolled screen.

- **Match the app's number formatting.** Don't hardcode `"$400.00"`-style strings in
  replicas — pull amounts through the app's own formatter / `MoneyText`. (Kachak's
  `#,##0.##` drops trailing zero cents: `$708.1 / $600`, and positive net shows no `+`.)

- **Full-bleed white background inside the bezel**: a `Column` inside `Positioned.fill`
  must have `crossAxisAlignment: CrossAxisAlignment.stretch`, or a `Container(color:
  white)` shrinks to wrap its text content instead of filling the bezel width.

- **Mini thumbnail of another screen**: render the other screen's content at a small
  fixed size (e.g. 52×104 dp) using the same structure as the full screen but with
  proportionally tiny font sizes (3–6 dp). Position it via `Positioned` inside the
  composite screen's `Stack`.

- **Stacked OS notification cards**: renders the capture story as two Android-style
  notification cards — source app (a fictional bank, e.g. Cat Bank — see §9 brand
  safety) below, your app (Kachak) above. Each card: white,
  `BorderRadius.circular(16)`, drop shadow, app icon in a rounded square, app name +
  timestamp in the header row, title + body below.

- **Modal scrim**: inside the screen body (not the outer Stack), add a
  `Positioned.fill(child: ColoredBox(color: Color(0x72000000)))` as the last child of
  the body's Stack. This dims the settings content exactly like a real modal, while the
  hero dialog (in the outer Stack) sits above it naturally.

- **App-picker dialog hero**: mirror the structure of the real app's AlertDialog —
  title, search field, `CheckboxListTile` list, Cancel/Save buttons. Key sizing:
  width ~80% of bezel (see §7), `dense: true` + `visualDensity: VisualDensity.compact`
  + reduced `contentPadding` on the list tiles to keep the dialog compact enough to show
  buttons. Trim the list to only the *checked* items so the buttons are visible; unchecked
  items add length without telling the story.


## 11. ★ The tuning playbook

These are the lessons that only come from looking at rendered pixels. They cannot be
calculated up front — they require the render → inspect → adjust loop.

**Hero positioning is hand-tuned, not formulaic.** Start with a fraction guess (0.0 for
"at the top", 0.5 for "centered", 0.6–0.7 for "near the bottom"). Render. Eyeball.
Nudge the fraction by 0.05–0.10 and re-run. The right value is "visually obvious that
the hero is the focal point."

**Converting user-space nudges to pre-scale offsets.** When the user says "move it down
10% of the bezel height":
- 10% of `bezelH` (672 dp in App Store space) = 67.2 dp canvas shift.
- Convert to pre-scale hero-internal offset: `67.2 / 1.15 ≈ 58 dp`.
- Add to the current `top` value inside the hero's inner Stack.

**Trimming > shrinking.** When a hero dialog is too tall and the action buttons fall off
canvas: remove list items (keep only the 3 most important ones) rather than compressing
font sizes or padding. The result looks cleaner and tells a clearer story.

**Dense rows for settings fakes.** Real settings screens look dense. Apply `dense: true`
+ `visualDensity: VisualDensity.compact` to both `SwitchListTile` and `ListTile` in any
mock settings screen; without it the tiles have 16 dp of vertical padding each and the
screen looks airy/fake.

**Dialog width = ~80% of bezel, centered.** A full-bezel-width "dialog" reads as a
bottom sheet or overlay, not a picker dialog. Cap it at ~80% and center it
(`Align(topCenter, child: SizedBox(width: 240))` pre-scale) for instant dialog recognition.

**Match the real app's labels exactly.** When faking a settings section, grep the real
source for the exact l10n strings and tile structure. Users recognize their own app — a
label that says "Auto-capture" when the real app says "Capture transaction notifications"
looks wrong even in a marketing image. Pull the real `SwitchListTile` secondary icon,
title, subtitle, and `ListTile` leading icon from the source.

**The `const` gotcha with SVG logos.** If the app uses `flutter_svg` for a logo widget,
it is not a `const` constructor. Removing `const` from its parent `Center()` (or
wherever it is nested inside a `const` expression) fixes the compile error.

**`KachakLogo` (or any svg widget) is not const**: remove `const` from ancestor
expressions that contain it.


## 12. Reproduction workflow

Order of operations for a new app:

1. **Bootstrap** — set up the test file: ffi init, channel mocks, SharedPreferences
   mock, repo open + currency init + category seed.
2. **Seed** — wipe + insert the demo persona into the in-memory DB.
3. **Define the frame list** — for each screenshot: a slug, a headline, an optional
   sub-caption, a screen builder (`() => const MyScreen()`), and a hero builder
   (`(ctx) => const _HeroMyCard()`). Specify `heroFraction` / `bezelTopOffset` if needed.
4. **Write `_ScreenshotFrame`** — gradient + caption + bezel + outer-Stack hero. Reuse
   the Apple-master / Play-shrink-fit architecture from §3.
5. **Write hero widgets** — one per frame; mock-only, no production code. Use theme
   tokens (`ColorScheme`, `KachakColors` extension) rather than hardcoded hex.
6. **Capture loop** — for each frame × each export config: set surface size, pump widget,
   wait for animations, capture `RepaintBoundary`, write PNG.
7. **Eyeball** — open every PNG. Fix icons (§8), underlines (§8), hero overlap issues
   (adjust fractions), caption overflow (reduce font size or add captionH).
8. **Tune** — one frame at a time, applying the playbook from §11. Re-run after each
   change.

One command regenerates all frames:
```
flutter test test/screenshots/screenshot_harness.dart --timeout=120s
```


## 13. Verification checklist

Before uploading to the stores:

- [ ] Correct file counts: `N` PNGs in `screenshots/appstore/` and `N` in `screenshots/play/`.
- [ ] Correct pixel sizes: `1290×2796` (App Store) and `1080×1920` (Play).
- [ ] No box-glyph icons (→ fix font loading, §8).
